## Part 2：10 個核心業務流程

每流程要回答：
- **誰起 → 誰接 → 誰結束**

### 2.1 批發訂單流程 ⭐ 最高頻

```
[業務員] 客戶詢盤
   ↓ 立即填單（lobby「+新訂單」）
[業務員 → 安然 + 賴哥 + 高維 + 躍明 同時 notify]
   ↓ 安然 24hr 內把關（IP 授權 + 客戶資質 + 工廠 capacity）
   ├─ 退回（需資訊）→ 通知業務員補
   ├─ 不接 → 通知業務員告知客戶
   └─ 可執行 → 轉 Ray
[安然 → Ray]
   ↓ Ray 24hr 內拍板（定價）
   ├─ 批准 → 通知業務員（報價給客戶）+ 賴哥（開 PO）+ 高維（預排出貨）
   ├─ 退回修改
   └─ 暫緩
[Ray → 業務員 / 賴哥 / 高維]
   ↓ 業務員報價客戶 → 客戶確認下訂金（30%）
   ↓ 賴哥開 PO → 工廠生產
   ↓ 木子核對廠端備貨
   ↓ 高維填單號出貨
   ↓ 客戶收貨 → 付尾款（70%）
[楊子/Leo（出納收款）→ 躍明（入帳）]
   ↓ 楊子確認款項收到 → 推給躍明
[躍明 → 入帳 → VAM 計入]
```

**SLA：**
- 新訂單填單：當天
- 安然把關：24 hr
- Ray 拍板：24 hr
- 賴哥開 PO：3 天內
- 木子備貨核對：到貨當天
- 高維出貨：備齊後 2 天
- 楊子/Leo 確認收款：當天
- 躍明入帳：3 天內

**卡住誰追：**
- 安然 > 24hr 沒批 → Shawn 知會（但 Shawn 不能替決策，等安然或 Ray）
- Ray > 24hr 沒拍板 → Shawn 知會（很罕見，Shawn 主動找 Ray）
- 賴哥 > 3 天沒開 PO → 安然 直接處理（Shawn 知會）
- 楊子 > 1 週沒批 → 躍明 + Shawn

### 2.3 供應商 PO 流程（採購）

```
[Ray 批准訂單後] 賴哥收到「開 PO」訊號
   ↓ 賴哥詢比價（如新供應商）/ 直接下單（既有供應商）

[木子 → 賴哥 + 高維（可出貨）]

   ↓ 同時，木子「付款申請」（給工廠尾款）
```

## Part 4：監督機制（Shawn = 透明度監督，不替代決策）

**重要原則：**
- Shawn 看到全公司所有卡住的事（透明度）
- Shawn **不能替代任何主管做決策**
- **安然 domain 的事**（商務 / 業務 / IP / 合約 / 跨境）→ Shawn 只能**催 + 知會 Ray**，不能升級替安然決策
- **Shawn 自己 domain**（高維 / 春雷 / 佳音 / 海外 / 快閃）→ Shawn 直接決策 + 升級 Ray
- 升級給 Ray 的權限按 domain 區分

### 4.1 Shawn 收到 4 種「卡住」訊號

| 訊號 | 偵測規則 | 範例 |
|---|---|---|
| **逾期未處理** | inbox item submittedAt > SLA 天且 status === pending | 楊子 5 天沒批費用 |
| **退回後沒補** | 被 reject 後 N 天當事者沒重交 | 業務員被退 3 天還沒補資訊 |
| **轉介後沒進度** | A 轉給 B 後 B 拿到 N 天沒動 | 安然轉珮綾 7 天珮綾沒追 |
| **任務逾期** | deadline 已過 + status !== done | PO 沒按時出 / 拜訪該填沒填 |

### 4.2 SLA 表（卡多久算逾期）

| Inbox 類型 | 一般 | 高優先 |
|---|---|---|
| 楊子/Leo 費用審核 | 3 天 | 24 hr |
| 安然合約 / 跨境 | 3 天 | 24 hr |
| Ray 批核 | 2 天 | 24 hr |
| Bevis 設計核准 | 3 天 | 24 hr |
| 珮綾送審追蹤（等 IP 方）| 14 天 | 7 天 |
| 賴哥 PO | 7 天 | 3 天 |
| 木子備貨核對 | 廠端通知後 1 天 | 當天 |
| 高維出貨 | 備齊後 2 天 | 當天 |
| 業務員拜訪填單 | 1 天 | — |

### 4.3 Shawn「待催辦」list 的真實邏輯

```javascript
// 系統掃描（每次 Shawn 進 lobby 觸發）
const stale = [];
['yangzi_inbox_v1','anran_contract_inbox_v1','anran_xborder_inbox_v1',
 'bevis_design_approve_v1','peilin_approval_track_v1','laige_po_v1',
 'ray_approval_v1','muzi_stock_check_v1','gaowei_logistics_v1', ...]
  .forEach(key => {
    const arr = JSON.parse(localStorage.getItem(key) || '[]');
    arr.forEach(item => {
      const ageDays = (Date.now() - item.submittedAt) / 86400000;
      const sla = SLA_MAP[key][item.urgency || 'normal'];
      if(!item.done && !item.status && ageDays > sla){
        stale.push({source:key, item, daysLate:Math.floor(ageDays - sla)});
      }
    });
  });
// stale 自動 populate Shawn 待催辦
```

### 4.4 Shawn 點開「待催辦」可做的動作（按 domain 區分）

**Shawn 自己 domain（高維 / 春雷 / 佳音 / 海外 / 快閃店）：**

| 動作 | 效果 |
|---|---|
| 🔔 **飛書催當事者** | push 飛書 + 寫 notif queue |
| ⬆️ **升級給 Ray** | 寫進 ray_approval_v1 + 通知 Ray |
| ✓ **暫不催（24 hr）** | mark seen，list 隱藏 24 hr |

**安然 domain（業務員 / 採購 / 楊子 / 合約 / IP / 跨境）：**

| 動作 | 效果 |
|---|---|
| 🔔 **飛書催當事者** | push 飛書（提醒安然 push 該員工）|
| 💬 **知會安然** | 寫進安然 notif（請她處理）— **不能直接升 Ray** |
| ✓ **暫不催（24 hr）** | mark seen，list 隱藏 24 hr |

**為什麼差別：**
- 安然跟 Shawn 平級。Shawn 不能替安然向 Ray 升級她管轄的事
- 安然 domain 的事 Shawn 看到，但決策權在安然
- 若 Shawn 認為事態緊急 → 知會安然 → 安然自己決定升不升 Ray

## gg 補充框架觀念（寫進 Shawn 監督機制）

- **Manage by Exception**：Shawn 的角色 = 例外管理。系統把 26 個 modal 中的正常流程隱藏，只把「逾期 / 退回沒補 / 卡住」端到他面前
- **Domain 邊界**：Shawn（營運）vs 安然（商務）平級 — Shawn 有透明度（可催）但無跨界決策權（不能替安然向 Ray 升級）
- **Alert Fatigue 防範**：所有 SLA 不能太敏感，避免警報疲勞讓員工亂填消通知
