## 0. 背景

蕪解 ERP 主系統功能架構已完成（16 個帳號工作台 + 飛書資料層 + 多主體架構），但 Ray 在實際 review 後反饋四個核心問題：

1. 對團隊多數人太複雜，需要安排培訓才能用
2. UX 不直覺，打開不知從何下手
3. 措辭充滿術語（戰情室／流水線／審批／VAM …）不夠白話
4. UI 沒設計感

而公司即將讓全員開始使用系統。Ray 認為：**只有真正用起來，才能知道哪裡還要調整。** 因此這次討論的目標是：

> **「上線當天，大家看到的頁面應該是什麼樣子？」**

並在上線之前，把模擬數據清掉，把過去散落在人腦袋裡的經驗結構化進系統。

## 一、識別的核心痛點（Ray 親述，按工作流順序）

Ray 描述的不是 16 個獨立角色，而是**一條 7 棒接力的項目生命週期**：

```
預算 → 提案 → 設計 → 採購比價 → 打樣 → 業務收單 → 量產出貨 → 結案覆盤
 ↑                                                                  ↓
 預估數字 ─────────────── 實際數字（要對得起來）──────────────────────
```

### 痛點 1：預算建立階段（Shawn 主責）

- **過去成本資料找不到 / 不確定是不是最新版**
  - 報價單散落在微信、LINE、本機
  - 人事變動導致報價單失傳
  - 財務有合約／報價單存檔，但不知道現在討論的是哪一份
- **建預算時合約還沒談妥**
  - 例：安然收到面包超人版權方資訊問有沒有興趣，但商務條件未定
  - 系統應允許「TBD 後補」：項目可先建，版權條件待簽約後補齊
- **收支時間節點不細**
  - 模具費通常 30%-70% 分兩期
  - 批發收入因客戶帳期不同（出貨收 vs Net30/60/90）拆比例
  - 預算時就要拆時間軸 → 才能讓財務做精準現金流預測
  - 還要疊加：每月固定費用、北京蕪解銀行貸款利息＋本金攤還

### 痛點 2：實際 vs 預算對不起來

- 預算做了，但後續實際收支沒 link 回項目
- 因此無法判斷項目達成率 vs 預算落差
- Shawn / 安然 / Ray / 躍明 都不清楚項目實際情況
- 只有快閃店因要對版權方結案才相對精準，但事後對帳整理花很多時間

### 痛點 3：預算 → 企劃／設計 之間斷鏈

- 管理層立了項目，企劃／創意部沒被即時通知
- 企劃只盯單一項目做，不知道週／月／季要完成多少
- 都是憑感覺，等 Ray 喊「太慢了該外包」
- 不是計劃性安排

### 痛點 4：產品開發進度透明度

- 安然 / Shawn / 各部門想知道進度，要私訊問產開
- 同樣問題被問好幾次
- 或開線上會議全員同步
- 共識會議後「誰盯誰負責」不清晰，權責不明

### 痛點 5：採購環節權責不清

- 詢價比價完成後，交誰確認不清晰（Shawn? Ray? 不一定）
- 確認後資訊沒同步財務，採購不一定能拿到資金
- 不會明確回覆「最終交給哪間供應鏈」

### 痛點 6：打樣後審核流程不完整

- 打樣應該先給「管理層 + 設計部 + 產開部」三方確認
- 但現況經常直接交設計部送版權，跳過內部審核

### 痛點 7：意向單 vs 真實訂單混淆

- 業務拿擬真效果圖 → 收意向單（客戶無責任）
- 業務拿到工廠實樣 + 正式可交貨時間 → 才是真實訂單
- 這是「**最後一次修正售價／暫停項目**」的機會（模具還沒開）
- 系統應該標清楚這兩種狀態，避免誤把意向當訂單

### 痛點 8：庫存掌握在人腦裡

- 批發業務剩多少庫存只有安然／銷售估算
- 用「總生產數 − 已賣出」自己推估
- 還要透過高維／採購跟工廠二次確認
- 沒有單一真相

### 痛點 9：時間倒推 + 早期預警缺失

- 預算建立時應該有「計劃發售日」
- 從發售日倒推每階段最晚截止日（模具 45 天、量產 60 天、送審 3-6 個月）
- 如果某階段拉緊 → 系統提早警示
- 警示同時給「決策選項」：加人 / 外包 / 砍款 / 延期，各帶成本估算
- 管理層批准方案後，自動派下一棒
- **過去：事後才發現拉緊，憑感覺處理。未來：系統提前算給選項。**

### 痛點 10：合約／報價單散落

- 想確認上次工廠報價，要問好幾個同事
- 每人都不確定自己手上是不是最新版
- 想確認合約多久到期、抽成多少、MG 用了多少 → 都憑記憶猜
- 沒有 100% 確認的依據

### 痛點 11：版權方送審節奏差異大

- 每個版權方回覆週期不同
  - Snoopy / Peanuts：每週
  - Mofusand Inc.：每兩週
  - Muziktiger / 佳盛：每月
- 這是合約屬性（簽約時就應該跟業務窗口確認）
- 倒推算法的關鍵變數

## 二、系統核心架構：「**項目（Project）為中心**」

### 設計哲學轉變

| | 過去（廳為中心）| 未來（項目為中心）|
|---|---|---|
| 入口邏輯 | 16 個角色看 16 個工作台 | 所有人看「我參與的項目」清單 |
| 資料組織 | 廳擁有資料（war room／採購台／業務 CRM 各自獨立）| 一切掛在項目下 |
| 報價單／合約 | 散在各表 | link 到項目 |
| 採購／銷售／費用 | 獨立記錄 | 必填「對應項目」 |
| 庫存 | 各倉各算 | 項目級總覽（工廠／倉／在途）|

### 項目的完整資料結構

每個項目（Project）包含：

```
項目基本資訊
├── 項目名稱
├── IP / 版權方（可 TBD）
├── 對應合約（可 TBD）
├── 產品類型（PVC 公仔 / 文創周邊 / 紙盒包裝 / ...）
├── 計劃發售日 ★必填
├── 預估數量 / 預估售價
└── 主負責人

時間表（系統倒推 + 人工微調確認）
├── 預算建立日
├── 企劃提案截止
├── 設計定稿截止
├── 送審通過預估
├── 打樣定稿截止
├── 模具完成截止
├── 量產完成截止
├── 物流到貨截止
└── 發售日 ★

預算（分批次 + 分時間節點）
├── 模具費（分階段：開模 30% + 驗收 70%）
├── 採購成本（30%-40%-30% 等）
├── 版權費用（MG + 抽成）
├── 行銷預算
├── 物流成本
└── 預估收入（按客戶帳期拆比例）

附件庫（link 在項目下）
├── 報價單版本（v1, v2, v3...）
├── 設計稿
├── 打樣照片
├── 送審記錄
├── 版權批准信
└── 樣品照

實際數（自動 link，計算達成率）
├── 實際採購支出
├── 實際銷售收入
├── 實際庫存
└── 預算 vs 實際差距

狀態機（7 階段 + 各階段 owner）
└── 進度 progress dots：預算 → 提案 → 設計 → 採購 → 打樣 → 收單 → 量產出貨
```

## 三、建立項目流程：3 段確認 UX

### 為什麼分 3 段

- 一次填太多欄位會嚇人
- 每段都讓人 review 系統預設值並可微調
- 確認後責任就在這一刻定下來

### 第 1 段：基本資訊

| 欄位 | 必填 | 備註 |
|---|---|---|
| 項目名稱 | ✓ | |
| IP / 版權方 | 可 TBD | 如已簽約，link 到合約；如未簽約，標 TBD |
| 對應合約 | 可 TBD | 同上 |
| 產品類型 | ✓ | SingleSelect |
| 計劃發售日 | ✓ | 倒推算法的終點 |
| 預估數量 | ✓ | |
| 預估售價 | ✓ | |
| 主負責人 | ✓ | |

### 第 2 段：時間表確認（系統倒推 + 微調）

**系統根據「產品類型 + IP 送審週期 + 物流路線」自動帶入預估天數，倒推每階段截止日。**

範例：發售日 2026-12-01

| 階段 | 預估天數 | 截止日 |
|---|---|---|
| 國際物流（北京→台）| 15 天 | 11/15 |
| 量產（PVC 公仔）| 60 天 | 09/16 |
| 模具 | 45 天 | 08/01 |
| 打樣定稿 | 14 天 | 07/17 |
| 送審（Mofusand × 3 輪 × 14 天）| 42 天 ⚠️ | 06/05 |
| 設計定稿 | 14 天 | 05/22 |
| 企劃提案 | 14 天 | 05/08 |
| 預算建立 | — | **今天** |

**緩衝天數警示：** 系統算出緩衝 8 天，建議至少 14 天 → flag 警示

**可微調：** 點任一階段可改天數，系統重算所有 deadline

### 第 3 段：收付款節奏確認

**採購付款拆解**

| 項目 | 比例 | 觸發點 |
|---|---|---|
| 模具費 | 30% | 開模時 |
| 模具費 | 70% | 模具驗收 |
| 量產費 | 30% | 下單 |
| 量產費 | 40% | 量產完成驗收 |
| 量產費 | 30% | 出貨後 |

**銷售收款拆解**

| 客戶類型 | 比例 | 帳期 |
|---|---|---|
| KA 客戶 | 70% | Net60 |
| 流通商 | 30% | 出貨收 |

確認後 → 項目建立完成 → 自動通知企劃部／採購／管理層

## 四、時間倒推 + 早期預警機制

### 倒推算法輸入

- 計劃發售日（項目屬性）
- 產品類型 → 模具天數、量產天數（系統常數）
- IP 對應合約 → 版權方送審週期 × 預估輪數
- 出口地 → 國際物流天數（系統常數）
- 緩衝天數（系統建議 ≥14 天）

### 預警觸發條件

當任一階段「實際進度 < 倒推 deadline」時，系統 surface 預警卡片給 Ray + Shawn。

### 預警卡片內容

```
⚠️ Mofusand Q4 聖誕款 進度落後 5 天

設計定稿原計劃 7/1，目前還在第 2 輪修改
風險：發售日 12/1 不變的話，後續每階段都要壓縮

建議方案（按成本／影響排序）：

| 方案 | 額外成本 | 影響 | 需誰同意 |
|---|---|---|---|
| A. 外包設計給 X 工作室 | +¥5 萬 | 7/3 可定稿 | Shawn |
| B. 內部設計師加班 | +¥1.5 萬 | 7/5 可定稿 | Bevis |
| C. 砍 1 款 SKU | -¥30 萬營收 | 7/1 可定稿 | Shawn + Ray |
| D. 延後發售至 12/8 | 錯過聖誕首檔 | Ray |
```

### 批准後系統自動：

- 更新項目預算（如選 A，加 ¥5 萬到設計費）
- 通知產開部「下一棒是 X 工作室，截止 7/3」
- 重新計算後續階段 deadline

## 五、經驗常數庫：用「結構化問卷」取代「上傳文件」

### Ray 的核心洞察

> **經驗在人腦袋裡，不在文件裡。**
> 合約 PDF 上傳了，但「Mofusand 第 3 輪通常卡比較久」「A 工廠最近搬廠加 5 天」這些**判斷力**沒在文件裡，是在賴哥、安然、Shawn 的腦袋裡。

### 為什麼問卷比文件好

| | 上傳文件 | 結構化問卷 |
|---|---|---|
| 阻力 | 高（要翻電腦／微信）| 低（憑經驗答）|
| 完整性 | 看誰存了什麼 | 系統定義必填欄位 |
| 可用性 | 還要解析才能用 | 直接成為系統參數 |
| 維護性 | 文件不會更新 | 問卷可定期 refresh |
| 心理 | 「找文件好麻煩」| 「我把經驗分享給系統了」（有 ownership 感）|

**文件變成「需要時才補」的選用功能（例：MG 攤提爭議時翻原合約）。日常運作靠常數庫。**
